# 代码编译测试工具 (Debug Tool)

NVMe SSD 固件编译、烧卡、测试一体化工具，支持 QA 脚本测试和自定义 FIO 性能测试，集成 LLM 智能分析。

## 一、环境配置

### 1.1 Windows 用户

**第一步：安装 Python**

1. 前往 [Python 官网](https://www.python.org/downloads/) 下载 Python 3.8 或更高版本
2. 安装时 **务必勾选 "Add Python to PATH"**
3. 安装完成后，打开命令提示符验证：
   ```
   python --version
   ```

**第二步：获取项目**

将项目下载或克隆到本地任意目录。

**第三步：一键启动**

双击项目根目录下的 `启动.bat`，脚本会自动：
- 检测 Python 是否安装
- 检测并安装依赖库（paramiko、requests、openai）
- 启动程序

> 如果双击后窗口一闪而过，说明 Python 未正确安装或未添加到 PATH，请重新安装并勾选 "Add Python to PATH"。

### 1.2 Mac 用户

**第一步：安装 Python**

打开终端（Terminal），执行：
```bash
# 检查是否已安装 Python 3
python3 --version

# 如果未安装，通过 Homebrew 安装
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
brew install python3
```

**第二步：获取项目**

```bash
git clone <项目仓库地址>
cd debug_tool
```

**第三步：安装依赖**

```bash
pip3 install -r requirements.txt
```

如果提示权限不足：
```bash
pip3 install --user -r requirements.txt
```

**第四步：启动程序**

```bash
python3 main.py
```

### 1.3 参数配置

首次运行程序后，会在项目根目录自动生成 `config.json` 文件，**必须修改以下配置才能正常使用**。

用文本编辑器打开 `config.json`，修改以下内容：

#### 编译服务器配置

编译服务器是执行固件编译的远程服务器。

```json
"build_server": {
    "host": "你的编译服务器地址",
    "port": 22,
    "username": "你的用户名",
    "password": "你的密码"
}
```

| 参数 | 说明 | 示例 |
|------|------|------|
| host | 编译服务器域名或IP | `BS_JUJUBE_SZ_01.memblaze.com` |
| port | SSH 端口 | `22` |
| username | 登录用户名 | `haodong.zhang` |
| password | 登录密码 | `your_password` |

#### 测试服务器配置

测试服务器是执行烧卡和测试的远程服务器。

```json
"test_server": {
    "host": "你的测试服务器IP",
    "port": 22,
    "username": "你的用户名",
    "password": "你的密码",
    "flash_script": "/path/to/flash_script.sh"
}
```

| 参数 | 说明 | 示例 |
|------|------|------|
| host | 测试服务器IP地址 | `172.16.8.51` |
| port | SSH 端口 | `22` |
| username | 登录用户名（与编译服务器通常相同） | `haodong.zhang` |
| password | 登录密码 | `your_password` |
| flash_script | 烧卡脚本路径（一般无需修改） | `/path/to/flash_script.sh` |

#### AI 模型配置

用于 LLM 智能分析（FIO 命令生成、错误分析）。可通过界面上的「配置AI模型信息」按钮修改，也可直接编辑 config.json。

```json
"llm": {
    "api_key": "你的API密钥",
    "model": "模型名称",
    "api_url": "API地址"
}
```

| 参数 | 说明 | 示例 |
|------|------|------|
| api_key | LLM API 密钥 | `sk-xxxxxxxx` |
| model | 模型名称 | `DeepSeek-V3.2` |
| api_url | API 地址 | `http://api.memblaze.com/v1` |

> **提示**：`username` 和 `password` 也可以在程序界面的「用户账号」输入框中填写，程序会自动更新 config.json 中编译服务器和测试服务器的用户名。但首次使用前，服务器地址和密码必须先在 config.json 中配置好。

## 二、使用教程

### 2.1 启动项目

- **Windows**：双击 `启动.bat`
- **Mac**：终端执行 `python3 main.py`

启动后出现主界面，分为三个区域：顶部「配置信息」、中部「操作按钮」、底部「执行输出」。

### 2.2 界面参数说明

#### 配置信息区域

| 参数 | 说明 | 示例 |
|------|------|------|
| 用户账号 | 编译/测试服务器登录用户名 | `haodong.zhang` |
| 编译路径 | 编译服务器上固件项目路径 | `/home/haodong.zhang/firmware/` |
| 测试机 | 被测设备主机名 | `szf48` |
| 项目名 | 固件项目名称（下拉选择） | `goji` 或 `baiji` |
| SKU | 产品容量规格（下拉选择） | `ut768` |
| SS开卡 | 是否使用 SS 开卡模式（勾选框） | 勾选/不勾选 |
| DDR | 内存类型（单选：DDR4/DDR5） | DDR4 |
| 测试模式 | QA脚本测试 或 自定义FIO测试 | 两种模式二选一 |
| 测试脚本 | QA测试脚本名称（仅QA模式显示） | `IoExerciser.py` |
| Case | 测试用例编号（仅QA模式显示） | `all` 或具体编号 |
| Other | 其他测试参数（仅QA模式显示） | `null` 或自定义参数 |
| 测试需求 | FIO测试需求描述（仅FIO模式显示） | `128k顺序读写测试` |
| 烧卡类型 | NCMT 或 FW Download | NCMT |
| 收集串口日志 | 是否在测试期间收集串口日志 | 勾选 |
| 上传测试结果 | 是否将测试结果上传到数据库 | 否 |
| 版本号 | 固件版本号（上传测试结果时需填写） | `1.0.0` |

#### SKU 参数拼接规则

最终传递给烧卡/测试命令的 SKU 参数由三部分拼接：

| SKU基础 | SS开卡 | DDR | 最终SKU参数 |
|---------|--------|-----|------------|
| ut768 | 不勾选 | DDR4 | `ut768-ddr4` |
| ut768 | 勾选 | DDR4 | `ut768-ss-ddr4` |
| ut768 | 不勾选 | DDR5 | `ut768-ddr5` |
| ut768 | 勾选 | DDR5 | `ut768-ss-ddr5` |

#### 操作按钮区域

| 按钮 | 说明 |
|------|------|
| 从步骤开始 | 选择从哪个步骤开始执行（编译/下载Output/上传Output/烧卡/测试） |
| 开始执行 | 开始执行工作流 |
| 停止执行 | 中途停止当前执行 |
| 配置AI模型信息 | 弹窗修改 LLM API 地址、密钥、模型名称 |

### 2.3 QA 脚本测试流程

1. 填写「用户账号」「测试机」「编译路径」「项目名」「SKU」等基本信息
2. 选择测试模式为「QA 脚本测试」
3. 在「测试脚本」输入框中输入或搜索脚本名称（输入时自动搜索匹配）
4. 如需运行多个脚本，点击 `+` 按钮添加更多脚本行，多个脚本将用 `&&` 串联执行
5. 选择「从步骤开始」的起始步骤
6. 点击「开始执行」

**完整流程**：编译 → 下载Output → 上传Output → 烧卡 → 测试

### 2.4 自定义 FIO 测试流程

1. 填写「用户账号」「测试机」「编译路径」「项目名」「SKU」等基本信息
2. 选择测试模式为「自定义 FIO 测试」
3. 在「测试需求」输入框中描述测试需求，例如：`128k顺序读写测试，4k随机读写测试，运行10分钟`
4. 选择「从步骤开始」的起始步骤（如果选择从编译开始，会先执行编译→烧卡流程，再执行FIO测试）
5. 点击「开始执行」
6. 程序调用 LLM 生成 FIO 命令，弹窗展示供确认
7. 可在弹窗中直接编辑修改命令内容
8. 确认后，程序自动 SSH 到测试机逐条执行 FIO 命令

## 三、功能手册

### 3.1 编译

**功能**：通过 SSH 连接到编译服务器，执行固件编译命令。

**编译命令自动生成**：根据「项目名」自动生成编译命令，格式为：
```
python3 /home/{用户名}/firmware/products/{项目名}/scripts/build_{项目名}_ymtc_blkXpage.py build release all
```

**编译结果判断**：
- 自动检测编译输出中的错误关键字（如 `error:`、`ninja: build stopped`、`subcommand failed`）
- 编译成功后自动获取 output 文件路径
- 编译失败时自动调用 LLM 分析错误原因

**注意事项**：
- 编译过程可能耗时较长（通常1-3分钟），请耐心等待
- 编译期间输出区域会实时显示编译日志，每行带有时间戳
- 可随时点击「停止执行」中断编译

### 3.2 下载/上传 Output

**下载 Output**：从编译服务器下载编译生成的 output 文件夹到本地 `./output` 目录。

- 下载前会自动清空本地旧的 output 目录
- 下载完成后会检查本地目录是否为空

**上传 Output**：将本地 output 文件夹上传到测试服务器的 `/home/{用户名}/firmware/output/` 目录。

- 上传前会自动删除测试服务器上的旧 output 目录
- 自动创建目录并设置权限

### 3.3 烧卡

**功能**：在测试服务器上执行烧卡脚本，将固件烧录到测试机。

**两种烧卡模式**：

| 模式 | 命令 | 说明 |
|------|------|------|
| NCMT | `Remanufacture.py --host={测试机} --sku={SKU} --product={项目名} --force --fwrev=...` | 标准烧卡模式，包含 SKU 参数 |
| FW Download | `FwDownload.py --product={项目名} --ca=3 --fs=2 --host={测试机} --fwrev=...` | 固件下载模式，不包含 SKU 参数 |

**烧卡结果判断**：
- 检查输出中 `OverallResult` 是否为 PASS
- 检查是否有 bash 错误或 command not found
- 烧卡失败时自动调用 LLM 分析错误

### 3.4 QA 脚本测试

**功能**：在测试服务器上执行 QA 测试脚本，验证固件功能。

**测试命令格式**：
```
{脚本名} --host={测试机} --sku={SKU} --product={项目名} --case={用例} {其他参数}
```

**多脚本执行**：
- 点击 `+` 按钮可添加更多脚本行
- 多个脚本使用 `&&` 串联，顺序执行
- 每个脚本可独立设置 Case 和 Other 参数
- 点击 `-` 按钮可删除对应脚本行（第一行不可删除）

**测试脚本搜索**：
- 在脚本输入框中输入时，自动从测试服务器搜索匹配的脚本
- 使用上下箭头键在搜索结果中移动
- 按回车键选择脚本

**测试结果判断**：
- 优先检查 `OverallResult` 标签（PASS/FAIL）
- 检查 RESULT 列中的 PASS/FAIL
- 检查是否有 Python 异常或 bash 错误

**上传测试结果**：
- 勾选「上传测试结果」后，测试命令会添加 `--upload-results-to-db` 参数
- 需要填写「版本号」，会作为 `--fwkey` 参数传入

### 3.5 自定义 FIO 测试

**功能**：通过 LLM 将自然语言测试需求转换为 FIO 命令，并在测试机上执行。

**详细流程**：

1. **需求输入**：在「测试需求」输入框中描述测试需求
2. **LLM 生成命令**：程序调用 LLM，参考内置的 FIO 命令模板生成标准 FIO 命令
3. **命令确认弹窗**：弹窗展示生成的 FIO 命令，用户可以：
   - 直接确认执行
   - 在文本框中编辑修改命令后再确认
   - 取消执行
4. **SSH 登录测试机**：程序自动从测试服务器 SSH 到测试机
   - 自动处理首次连接的 host key 确认提示
   - 自动输入密码完成登录
   - 支持免密登录
5. **逐条执行 FIO 命令**：每条命令执行完成后才执行下一条
6. **执行完成**：所有命令执行完毕后退出测试机

**FIO 命令模板**：

程序内置了两个 FIO 命令模板供 LLM 参考：

- **模板1 - 填盘命令**：128k 顺序写，iodepth=512，写满整个盘
- **模板2 - 性能测试**：128k QD16 顺序读，运行10分钟，输出 JSON 格式结果

**FIO 测试模式也支持完整流程**：

选择「从编译开始」时，会先执行编译→下载→上传→烧卡，然后再执行 FIO 测试。如果只需要执行 FIO 测试，选择「从测试开始」即可。

### 3.6 串口日志收集

**功能**：在测试期间通过 Uart.py 收集测试机的串口日志。

**工作方式**：
- 勾选「收集串口日志」后，在烧卡/测试步骤前自动启动串口收集
- 串口收集在独立线程中运行，不影响主流程
- 日志保存到 `logs/uart_log.txt`
- 串口收集会持续运行直到程序关闭

### 3.7 LLM 智能分析

**功能**：当编译、烧卡或测试失败时，自动调用 LLM 分析错误原因。

**分析场景**：
- **编译失败**：分析编译错误信息，定位错误原因
- **烧卡失败**：分析烧卡输出，判断失败原因
- **测试失败**：分析测试日志，结合串口日志给出分析

**分析结果展示**：
- 弹出错误弹窗，上方显示原始错误信息（红色）
- 下方实时流式显示 LLM 分析结果
- 分析过程中可以阅读已有内容

### 3.8 AI 模型配置

**功能**：在界面中直接修改 LLM 的 API 配置，无需手动编辑 config.json。

**操作方式**：
1. 点击主界面「配置AI模型信息」按钮
2. 在弹窗中修改 API 地址、API 密钥、模型名称
3. 点击「保存」，配置立即生效并写入 config.json

**常见模型配置**：

| 配置项 | 说明 |
|--------|------|
| API 地址 | LLM 服务的 API 端点，如 `http://api.memblaze.com/v1` |
| API 密钥 | 调用 API 的认证密钥 |
| 模型名称 | 使用的模型，如 `DeepSeek-V3.2`、`deepseek/deepseek-r1` |

### 3.9 界面配置记忆

**功能**：自动保存和恢复界面上的所有输入参数，下次打开程序时自动填充。

**自动保存的参数**：
- 用户账号、测试机、编译路径、项目名
- SKU、SS开卡、DDR 选择
- 测试脚本、Case、Other 参数（包括多行脚本）
- 测试模式、烧卡类型
- 收集串口日志、上传测试结果、版本号
- FIO 测试需求

**保存位置**：`ui_config.json`（项目根目录，已加入 .gitignore 不会上传到仓库）

**保存时机**：每次点击「开始执行」时自动保存

### 3.10 从任意步骤开始执行

**功能**：支持从工作流的任意步骤开始执行，避免重复执行已完成的步骤。

**可选起始步骤**：

| 起始步骤 | 执行内容 |
|----------|----------|
| 编译 | 编译 → 下载Output → 上传Output → 烧卡 → 测试 |
| 下载Output | 下载Output → 上传Output → 烧卡 → 测试 |
| 上传Output | 上传Output → 烧卡 → 测试 |
| 烧卡 | 烧卡 → 测试 |
| 测试 | 仅执行测试 |

**适用场景**：
- 编译已完成，只需重新烧卡测试 → 从「烧卡」开始
- 已烧卡成功，只需重新运行测试 → 从「测试」开始
- 本地已有 output 文件，只需上传烧卡 → 从「上传Output」开始

### 3.11 停止执行

**功能**：中途停止当前正在执行的工作流。

**操作方式**：点击「停止执行」按钮

**停止行为**：
- 向当前正在执行的远程命令发送 Ctrl+C 信号
- 关闭 SSH 通道
- 停止串口日志收集
- 界面恢复为可操作状态

### 3.12 追溯版本（二分法定位问题Commit）

**功能**：通过二分法在 git 提交历史中定位引入问题的具体 commit。适用于"某个版本的固件出现了性能跌落/功能异常，需要找到是哪一次提交引入的问题"这类场景。

#### 启用追溯版本

1. 在主界面勾选「追溯版本」复选框
2. 界面自动切换为追溯模式：
   - 「上传测试结果」和「版本号」行隐藏
   - 显示 **Start(新)** 和 **End(旧)** 两个 commit 下拉框
   - 程序自动从编译服务器获取最近 100 条 commit 记录填充下拉框
3. 取消勾选后恢复原状

#### 参数说明

| 参数 | 说明 |
|------|------|
| Start(新) | 较新的提交（git log 列表靠前，index 小），**已知不符合预期**的提交 |
| End(旧) | 较旧的提交（git log 列表靠后，index 大），**已知符合预期**的提交 |

**方向约束**：Start 必须比 End 新（index 更小）。选择 Start 后，End 下拉框会自动过滤，只显示 Start 及更旧的提交。

#### SKU 拼接规则（追溯模式下）

追溯模式的 SKU 由三部分拼接：

```
{基础SKU}{-ss(如果勾选)}{-ddr4或ddr5}
```

例如：`ut768-ss-ddr4`

#### 手动追溯流程

1. 填写所有必要参数（用户账号、测试机、项目名、SKU 等）
2. 选择 Start 和 End commit 范围
3. 点击「开始执行」→ 程序自动计算中间 commit → git reset 到中间 commit → 编译 → 烧卡 → 测试
4. 首次测试完成后，按钮变为三个选项：

| 按钮 | 含义 | 操作 |
|------|------|------|
| 向前追溯 | 问题在更旧的提交 | end = 中间commit，向旧方向搜索 |
| 向后追溯 | 问题在更新的提交 | start = 中间commit，向新方向搜索 |
| 结束追溯 | 结束追溯流程 | 恢复原状 |

5. 根据测试结果判断方向，反复迭代直到定位到目标 commit

#### 二分法原理

假设 100 条 commit，start=commit#1(最新/有问题)，end=commit#100(最旧/正常)：

```
第1轮: 测试 commit#50 → 不符合预期 → start=#50, end=#100
第2轮: 测试 commit#75 → 符合预期   → start=#50, end=#75
第3轮: 测试 commit#62 → 不符合预期 → start=#62, end=#75
...持续缩小范围直到相邻...
收敛: start=#68, end=#69 → 验证end(#69)
       #69符合预期 → 问题commit是#68
       #69也不符合预期 → 问题commit是#69
```

#### 注意事项

- 追溯模式下每次测试都会 `git reset --hard` 到指定 commit，会修改编译服务器上的代码状态
- 追溯完成后建议手动 `git checkout` 回到需要的分支
- 执行过程中可随时点击「停止执行」中断
- 配置记忆支持保存 Start/End commit 的选择

### 3.13 AI托管追溯

**功能**：在手动追溯的基础上，利用 LLM 自动判断每次测试结果是否符合预期，并自动决定追溯方向，实现全自动化二分法定位。

#### 使用步骤

**第一步：完成首次测试**

1. 正常使用追溯版本功能，从编译开始执行首次测试
2. 测试完成后，按钮区出现「向前追溯」「向后追溯」「AI托管」「结束追溯」

**第二步：配置 AI 托管**

1. 点击「AI托管」按钮，弹出二级配置窗口
2. 在 **Test Log 摘取** 区域：
   - 查看日志总行数提示
   - 输入起始行和结束行号，点击「提取」预览内容
   - 反复调整直到截取到包含关键测试数据的日志片段
3. 在 **期望判断依据** 文本框中输入判断标准，例如：
   - `我期望顺序读性能达到 12G/s`
   - `顺序写带宽应不低于 5G/s`
4. 点击「LLM分析」，让 LLM 试分析一次
5. 查看 LLM 分析结果，确认 LLM 能正确判断测试结果是否满足预期
6. 如果 LLM 判断准确 → 点击「开始托管」

**第三步：全自动追溯**

点击「开始托管」后，程序进入全自动循环：

```
┌─────────────────────────────────────────────┐
│  计算 start/end 的中间 commit                 │
│         ↓                                    │
│  git reset → 编译 → 烧卡 → 测试              │
│         ↓                                    │
│  从 test_log 截取指定区域                     │
│         ↓                                    │
│  发送给 LLM 判断: 符合预期?                   │
│    ├─ 不符合预期 → start = middle, 继续       │
│    └─ 符合预期   → end = middle, 继续        │
│         ↓                                    │
│  相邻 commit 时验证 end                       │
│    ├─ end 符合预期 → 报告 start 为问题commit  │
│    └─ end 不符合预期 → 报告 end 为问题commit  │
└─────────────────────────────────────────────┘
```

整个过程无需人工干预，直到找到最初出现问题的 commit。

#### AI托管的智能容错机制

| 场景 | 处理方式 |
|------|----------|
| 用户指定的日志区域信息不足，LLM 无法判断 | 自动使用完整日志重试 |
| 完整日志仍无法判断 | 停止托管，回到手动模式 |
| LLM API 调用失败 | 停止托管，回到手动模式 |
| git reset 失败 | 停止托管，回到手动模式 |
| 收敛到同一 commit | 自动报告结果并弹窗提示 |
| 用户中途停止 | 立即停止，不丢失已完成的进度 |

#### 弹窗行为差异

| 模式 | 测试完成时 |
|------|-----------|
| 手动追溯 / 普通 QA / FIO 测试 | 每次测试都弹出完成提示 |
| AI 托管 | 中间过程不弹窗，仅在最终定位到目标 commit 时弹出 🎯 结果提示 |

#### Prompt 设计

LLM 判断 prompt 包含以下要素：

```
【测试结果判断】

以下是测试日志的摘取片段（第X行到第Y行，共Z行）：
{提取的test_log内容}

用户的期望判断依据：{用户输入的期望}

请严格按照以下格式输出：
1. 判断结果：[符合预期/不符合预期/无法判断]
2. 分析依据：[引用日志中的关键数据]
3. 详细说明：[具体分析]
```

程序通过正则表达式精确提取 `判断结果` 字段的值作为判断依据，避免 LLM 输出中的干扰文本影响判断。

#### 典型使用场景示例

> **场景**：发现最新的 commit 顺序读性能有跌落，但一个月前的版本是正常的。

1. Start 选为最新 commit（有问题）
2. End 选为一个月前的 commit（正常）
3. 点击「开始执行」进行首次测试
4. 首次测试完成后点「AI托管」
5. 在弹窗中截取顺序读相关的日志区域（如 FIO 输出中的 read bw 数据）
6. 期望填入：「我期望顺序读性能达到 12G/s」
7. 点「LLM分析」验证 LLM 能正确识别性能数据
8. 点「开始托管」，程序自动循环直到找到跌落发生的 commit

## 常见问题

### Q: Windows 双击启动.bat 后窗口一闪而过

Python 未安装或未添加到 PATH。重新安装 Python 时务必勾选 "Add Python to PATH"。

### Q: Mac 提示 "Python not found"

Mac 系统自带 Python 2，需要使用 `python3` 命令：
```bash
python3 main.py
```

### Q: pip install 报错

尝试升级 pip 后重试：
```bash
python3 -m pip install --upgrade pip
pip3 install -r requirements.txt
```

### Q: 上传测试结果到网页版pbdt失败

检查pbdt服务器中个人pbdt文件夹中是否配置.apikey文件

### Q: SSH 连接失败

1. 确认 config.json 中服务器地址和端口正确
2. 确认用户名和密码正确
3. 确认网络可以访问服务器（可在终端测试：`ssh 用户名@服务器地址`）

### Q: LLM 调用失败（503 错误）

模型名称可能已变更，点击「配置AI模型信息」按钮更换可用的模型名称。

### Q: 首次 SSH 连接测试机卡住

程序已自动处理首次连接的 host key 确认提示，会自动发送 `yes`。如果仍然卡住，请检查测试机是否可达。

### Q: config.json 在哪里

位于项目根目录下，首次运行程序后自动生成。Windows 用户可在项目文件夹中找到，Mac 用户可通过终端查看。

### Q: 编译命令和项目名不匹配

编译命令根据「项目名」自动生成。例如项目名为 `baiji`，则编译命令为 `build_baiji_ymtc_blkXpage.py`。请确保项目名选择正确。
